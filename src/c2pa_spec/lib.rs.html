1//! Rust types for the [C2PA] core specification, generated from the
2//! [upstream CDDL schemas][schemas-zip] by the `c2pa-spec-codegen`
3//! workspace tool and committed in place.
4//!
5//! C2PA defines a CBOR data model for provenance and authenticity
6//! metadata on media assets. This crate is that model, in Rust.
7//!
8//! # What's inside
9//!
10//! The generated types cover the spec's assertions (actions, ingredient,
11//! hashes, soft binding, asset type, training and mining, etc.), the
12//! claim and manifest structures, hashed and external URI references,
13//! validation results and status codes, and the shared metadata and
14//! choice enums the other items reference. What's actually in the crate
15//! is whatever shows up below in the module type listing. It tracks the
16//! CDDL directly, so the set grows and shrinks with the upstream spec.
17//!
18//! Every generated type derives [`Clone`], [`Debug`],
19//! [`serde::Serialize`], and [`serde::Deserialize`], so values
20//! round-trip through any serde-compatible format. CBOR via
21//! [`ciborium`] is the wire format. CDDL comments on rules and fields
22//! carry over as Rust doc comments on the items they describe.
23//!
24//! Alongside the generated types:
25//!
26//! - [`valid_metadata_fields`]: valid XMP, Exif, IPTC, TIFF, PLUS, etc.
27//!   metadata field names from the spec's `valid_metadata_fields.yml`,
28//!   as string-slice constants.
29//! - [`urn`]: a parser for `urn:c2pa:` identifiers, matching the
30//!   grammar in `c2pa_urn.abnf`.
31//! - [`jumbf_uri`]: a parser for `self#jumbf=...` references. Wherever
32//!   the CDDL uses `jumbf-uri-type`, the generated field holds the
33//!   parsed type from this module instead of a raw `String`, so callers
34//!   can reach an embedded URN without parsing the string themselves.
35//!
36//! # The doc comments are non-normative
37//!
38//! The `///` text on generated types and fields is copied verbatim from
39//! the CDDL. Treat it as a reminder of what a field is for, not a
40//! precise definition. For authoritative semantics like value ranges,
41//! version-dependent behavior, cross-field constraints, or deprecation
42//! rules, go read the [C2PA technical specification][C2PA]. When the
43//! docstring and the spec disagree, the spec is right.
44//!
45//! # How the types are generated
46//!
47//! The CDDL files in `cddl/`, `valid_metadata_fields.yml`, and
48//! `c2pa_urn.abnf` are committed copies of the schemas that ship in the
49//! C2PA [schemas bundle][schemas-zip]. The `c2pa-spec-codegen` binary
50//! concatenates the CDDLs, parses them, and writes
51//! `c2pa-spec/src/generated.rs` plus `c2pa-spec/src/valid_metadata_fields.rs`.
52//! Running it with `--download` fetches a fresh bundle first. Regenerate
53//! with:
54//!
55//! ```text
56//! cargo run -p c2pa-spec-codegen -- --download
57//! ```
58//!
59//! [C2PA]: https://c2pa.org
60//! [schemas-zip]: https://spec.c2pa.org/specifications/specifications/2.4/specs/_attachments/C2PA_Schemas.zip
61//! [`ciborium`]: https://docs.rs/ciborium
62
